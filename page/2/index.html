chapter5_estimated

  • 预估难免不准确，记录下来，多次以后就会更加准确

  • 活动的规模
    1.活动预计要7个番茄钟，应该把这个活动拆开成多个活动，各自标注一个预估值。活动越大，预估越不准
    2.活动少于一个番茄钟，写入活动清单，在旁边标注一个0，然后在在今日待办中把多个这样的活动组合成一个活动

  • 选择
    1.每天为活动做预估，需要几个番茄钟就在活动旁边画上几个方格
    2.如果方格最终填满了或者是没填满，在清单里划掉
    3.如果活动在预估番茄钟数里没有做完，那么进行二次预估，用圆形表示，如果还是没有完成，用三角形进行三次预估，如果还没完成，你该反思了，试着将活动拆开，降低复杂度

  • 记录预估
    新增两列，第一列记录变化的方格之和，多出的用+，少的就用-，第二列记录重新预估的次数，第一列应该总是为0

chapter4_break

  • 切换心理状态
    休闲->战略->工作->休闲->战略->工作

  • 内部中断：自己造成的
    1.如果是突然想到的其他任务，那么把它记录到活动清单中去，必要的话加上最后期限，并记录下这次中断。可以用 “/” 符号
    2.如果是大型工作，比如忍不住上了个厕所，那么就把这个番茄钟作废，重新开始一个番茄钟，不可以把几段的时间加起来算一个番茄钟。可以记录在一个番茄钟开始多久后遇到中断

  • 外部中断：别人造成的
    1.电子邮件，信息之类，先不管他，这些应该放在一个番茄钟内进行
    2.同事的不紧急的活动，可以推迟的推迟，不要直接处理
    3.记录：记下中断类型，“-” 符号

  • 认清界限
    1.记录下来每天遇到的计划外活动有多少，可以评价自己的计划能力
    2.今日待办的事一定要做完，否则你就没有承诺感，无法起到激励的效果

chapter3_method

  • 你要准备什么：
  1. 活动清单：记录你最近要做的工作，不用排序，可以用很久
  2. 今日待办：今天要做的事，一天一新
  3. 记录：一天结束时收集到的数据，可以用很多天
  4. 计划外紧急：突然遇到的看似紧急的时，可以记录在今日待办下面
  • 一天的5个阶段
  1. 计划：在一天的开始，从活动清单中提取最重要的活动到今日待办表格中
  2. 跟踪：收集番茄钟过程中的过程指标，比如中断次数等等
  3. 记录：从原始的数据中提取有用信息
  4. 可视化：将信息组织起来，找出改进的思路

每天开始时进行计划，每天结束时，进行 记录，处理 和 可视化，中间进行周期性的 跟踪。

  • 与戴明循环思路相通

  • 工具
    铅笔，定时器，三张白纸。
    其中定时这个动作可以培养条件反射
    但是我喜欢用trello这个软件

  • 寻找起点
    选出最重要的活动，一个番茄钟后可以评估是不是最重要的。将 必须得做完 换成 从哪里开始 ，将 这个项目很大很重要 换成 我可以走出一小步 ， 每天给自己订的目标要合适合理

  • 技巧
    为了避免一头扎进不太重要的活动，我们要定期评估活动的重要性，还要频繁的休息，时不时自我评估，极力扬长避短。在清单上的活动可以使用荧光笔标注，可以加强心里印象。

  • 休息
    一个番茄钟25分钟后，进入休息时间后在表格后画一个X，休息3~5分钟，离开桌子，不要做重要的事，长期的精神会导致工作记忆能力和集中力下降，从而使工作快乐变成焦虑，灵感变成敏感，还容易发怒。休息后要决定时继续同一项活动，还是继续下一项活动，原因是重要性或者是活动已经完成。

  • 阶段性休息
    四个番茄钟后开始阶段休息，15~30分钟，清理办公桌，查看邮箱，但是不要回，要回复的请列入计划

  • 休息补充
    5分钟内该做的：睡觉，溜达，喝水，看窗外
    25分钟内该做的：交谈，看新闻，回电邮，工作
    如何睡觉，1）舒适的椅子2）一开始就闭上眼睛3）想象光扫描仪4）想象白色矩形5）休息完轻轻睁开眼睛

  • 完成的活动
    不要再番茄钟进行时切换到另一项活动，剩下的时间用来“过度学习”
    不要在番茄钟进行时休息，这会破坏效率

  • 持续改善
    1.保持简洁应该成为日常工作的一部分，你应该经常问自己，我是否需要让活动更小型化？我是否遇到经常性的干扰？是否有不必要的开销？
    2.制定评估标准，将测试指标和评估标准进行比较，分析，改进

  • 此时此地
    将活动按场合归类，比如食堂，宿舍，超市
    每天要有一个承诺：一个番茄钟，一个活动，一个目标

chapter1_onething

  • 番茄工作法概要：列出当天要做的事，，设置25分钟闹钟，然后重第一件事开始。另外加上每日回顾，每日承诺，控制中断，预估要花的时间等。

  • 为什么要使用
    1). 小事忙一天，大事一件没干：懒
    2). 面对复杂，望而却步：懒
    3). 一错再错，不长记性：懒
    4). 头脑被各种想法占据：想多了
    5). 把预估当成承诺：较真
    6). 没想到一件事要做怎么久：天真
    7). 没想到一件事怎么复杂：天真
    8). 休息到工作，心智调整不过来：懒

十条不错的编程观点

你越早开始都手编码，你所花费来编程的时间就越长 — Roy Carlson, University of Wisconsin

原型的价值在于他给你的教训，而不是代码自身 — Alan Cooper, software author, from The Inmates are Running the Asylum

  • 十条不错的编程观点
    转自陈皓的博客

1） The only “best practice” you should be using all the time is “Use Your Brain”.
唯一的“Best Practice”并不是使用各种各样被前人总结过的各种设计方法、模式，框架，那些著名的方法、模式、框架只代码赞同他们的人多，并不代表他们适合你，你应该更多的去使用你的大脑，独立地思考那些方法、模式、框架出现的原因和其背后的想法和思想，那才是“best practice”。事实上来说，那些所谓的“Best Practice”只不过是限制那些糟糕的程序员们的破坏力。

2）Programmers who don’t code in their spare time for fun will never become as good as those that do.
如果你对编程没有感到一种快乐，没有在你空闲的时候去以一种的娱乐方式去生活，无论是编程，还是运动，还是去旅游，那么你只不过是在应付你的工作，无时无刻不扎在程序堆中，这样下来，就算是你是一个非常聪明，非常有才华的人，你也不会成为一个优秀的编程员，要么只会平平凡凡，要么只会整天扎在技术中成为书呆子。当然，这个观点是有争议，热情和能力的差距也是很大的。不过我们可以从中汲取其正面的观点。

3）Most comments in code are in fact a pernicious form of code duplication.
注释应该是注释Why，而不是How和What，参看《惹恼程序员的十件事》，代码告诉你How，而注释应该告诉你Why。但大多数的程序并不知道什么是好的注释，那些注释其实和code是重复的，毫无意义。

4）XML is highly overrated
XML可能被高估了。XML对于Web上的应用是不错的，但是我们把其用到了各种地方，好像没有XML，我们都不会编程了。

5）Not all programmers are created equal
这是那些junior经理或是流程爱犯的错，他们总是认为，DeveloperA == DeveloperB，只要他们的title一样，他们以为他们的能力、工作速度、解决问题的方法，掌握的技能等等都是一样的。呵呵。更扯的是，在某些时候，就算是最差的程序员，他们也会认为其比别人强十倍，这就是现代的SB管理。

6）”Googling it” is okay!
Google只会给你知识，并不会教给你技能。那里只有“鱼”，没有“渔”，过度的使用Google，只会让你越来越离不开他，你越来越去要去立马告诉你答案，而你越来越不会自己去思考，自己去探索，去专研。如果KFC快餐是垃圾食品对我们的身体没有好处，那么使用Google也一种快餐文化对我们的智力发展大大的没有好处。

7）If you only know one language, no matter how well you know it, you’re not a great programmer.
如果你只懂一种语言，准确的说，如果你只懂一类语类，如：Java和C#，PHP和Perl，那么，你将会被局限起来，只有了解了各种各样的语言，了解了不同语言的不同方法 ，你才会有比较，只有了比较，你才会明白各种语言的长处和短处，才会让你有更为成熟的观点，而且不整天和别的程序在网上斗嘴争论是Windows好还是Unix好，是C好还是C++好，有这点工夫能干好多事了。世界因为不同而精彩，只知道事物的一面是有害的。

8）Your job is to put yourself out of work.
你的工作不是保守，那种教会徒弟，饿死师父的想法，不但是相当短浅的，而且还是相当脑残的。因为，在计算机世界里，你掌握的老技术越多，你就越没用，因为技术更新的太快。你对工作越保守，这个工作就越来越离不开你，你就越不越不能抽身去学新的东西，你也就越来越OUT了。记住：If you can’t be replaced then you can’t be promoted!

9）Design patterns are hurting good design more than they’re helping it.
很多程序员把设计模式奉为天神，他们过度的追求设计模式以至都都忘了需求是什么，结果整个系统设计被设计模式搞得乱七八糟，我们叫这种编程为“设计模式驱动编程”，正如第一点所说，如果你不懂得用自己的大脑思考的话，知其然，不知所以然的话，那么你不但得不到其好处，反而受其所累。

10）Unit Testing won’t help you write good code
准确地说，我们可以认为这是Test-Driven开发，其实，这种开发就是先写unit test case，这样的开发方式的主要目的是，为了防止你不会因为一个改动而引入Bug，但这并不会让你能写出更好的代码。这只会让你写出不会出错的代码。同第一点，这样的方法，只不过是防止糟糕的程序员，而并不是让程序员或代码质量更有长进。反而，通过Unit Test会为程序员的为自己代码做辩解的一种托辞。
最后，顺便说一下，以前去那个敏捷的公司面试，发现那个公司的某些技术人员中毒不浅，具体表现在上述的1）9）10）观点上。

android:style_theme

  1. style说指用来指定view和windows的样式，可以写在xml文件中

    <TextView
android:layout_width="fill_parent"
android:layout_height="wrap_content"
android:textColor="#00FF00"
android:typeface="monospace"
android:text="@string/hello" />

  2. 但是你也可以这么写，这样是为了重复使用这些代码，适当的使用回报是巨大的

    <TextView
 style="@style/CodeFont"
 android:text="@string/hello" />

  3. theme说适用与整个Activity或者application， 和view这样的Activity中独立的不宜眼，如果你把style可以用成theme，这样的话Activity中的所有view都会变成你设置的style，所以them就是一个全局的style

  4. 在res/resourse文件下建立一个.xml文件

    <?xml version="1.0" encoding="utf-8"?>
<resources>
<style name="CodeFont"       parent="@android:style/TextAppearance.Medium">
<item name="android:layout_width">fill_parent</item>
<item name="android:layout_height">wrap_content</item>
<item name="android:textColor">#00FF00</item>
<item name="android:typeface">monospace</item>
</style>
</resources>

用 @style/CodeFont就可以引用到这个style
parent=这个属性是可选的，表示继承那个属性，你也可以自己写一个继承，但是继承你自己写的style使用的方法不一样，不能用parent，而是使用 .,一下代码表示Red继承自CodeFont，你也可以多个继承<style name="CodeFont.Red.Big">但是注意，这个说自定义时使用的，要和parent分开

<style name="CodeFont.Red">
<item name="android:textColor">#FF0000</item>
</style>

  1. 对于使用的属性，要看你用的什么view了，有一些是通用的，比如android:layout_width，有一些是独特的android:inputType，这个属性应该放在<EditText />里面，具体可以到api里面查。另外大部分情况下style可以用作theme，但是theme有一些独特的属性，比如windowNoTitle and windowBackground

  2. 如何使用：style用在.xml文件里

    <TextView
style="@style/CodeFont"
android:text="@string/hello" />

    theme用在AndroidManifest.xml里

    <application android:theme="@style/CustomTheme">

    对于viewgroup里的view， style也是要一个一个指定的

  3. theme用在AndroidManifest.xml里

    <application android:theme="@style/CustomTheme">

    andorid有很多已经做好的主题，比如像对话框一样的

    <activity android:theme="@android:style/Theme.Dialog">

    比如背景透明的

    <activity android:theme="@android:style/Theme.Translucent">

  4. 你可以进行改动，加上自己喜欢的

    <color name="custom_theme_color">#b0b0ff</color>
<style name="CustomTheme" parent="android:Theme.Light">
<item     name="android:windowBackground">@color/custom_theme_color<   /item>
<item     name="android:colorBackground">@color/custom_theme_color    </item>
</style>

    注意这里的android:windowBackground使用了引用，因为这个属性只能提供引用，所以你要在另一个文件里制定，然后引用过来
